# fssstack

This is the **flatpack repo**: a source kit that an AI coding agent uses to create an application repo somewhere else. It is not an application repo itself, and it is not a git repo.

The separate empty repository where the application is assembled is the **target repo**.

The expected flow is:

1. Start with an empty target repo.
2. Run an AI coding agent from the target repo.
3. Have the agent read `SETUP_PROCESS.md` and follow it.

During setup, the agent reads source files from doctrine with commands such as `dx read`, generates generic frontend scaffolds with the shadcn CLI, installs dependencies, renders project/service-specific placeholders, and validates the result. The output is a runnable pnpm monorepo in the target repo; the flatpack repo is only the input used to assemble it.

The flatpack repo has a small tooling project for TypeScript setup scripts. You may run `pnpm lint`, `pnpm test`, `pnpm typecheck`, and `pnpm build` to validate that tooling. Do not run, lint, build, or test the flatpack repo as if it were the generated application. Validate the generated project from the target repo after setup is complete.

## How To Work In This Repo

Treat this repo as an assembly source kit. Most changes should affect one of these surfaces:

- Setup instructions in `SETUP_PROCESS.md`.
- Generated target files under `layers/`.
- Built setup scripts in `scripts/*.mjs`, with TypeScript sources in `scripts-src/`.
- Shared setup-script helpers in `scripts-src/lib/`.
- Extension instructions in `extensions/`.

When changing setup scripts, edit the TypeScript source in `scripts-src/`, then run `pnpm build` so the checked-in `scripts/*.mjs` artifacts are regenerated. Target repos execute the built `.mjs` files through `dx read`; if you only change `scripts-src/`, setup will not change for generated projects.

For example, to make `install-foundation` copy another root file into a target repo:

1. Add the source and destination pair to `foundationFiles` in `scripts-src/install-foundation.ts`.
2. Run `pnpm build`.
3. Confirm the same pair appears in `scripts/install-foundation.mjs`.

Use `pnpm lint`, `pnpm typecheck`, and `pnpm test` when changing script logic or helpers. For a narrow file-list change, `pnpm build` plus inspection of the generated artifact is usually the minimum useful validation.

## Markdown And Process Changes

Markdown in this repo is usually executable guidance for a future agent, not descriptive prose for this repo alone. Before changing setup or extension markdown, reason through the target repo that will exist when the instruction runs:

- What files already exist at that stage of `SETUP_PROCESS.md`?
- Which files come from `layers/`, which are generated by external CLIs, and which are rendered from template tokens?
- Is the command run from the empty target repo, from a package directory, or from this flatpack repo?
- Does the instruction rely on local flatpack files? If so, change it to use `dx read` or generated target paths instead.
- Does the generated project still validate from the target repo after the instruction changes?

Use `TARGET_PROJECT_SHAPE.md` as a reference for how a generated target repo unfolds across the setup stages. Keep that document current when changing the setup flow in a way that affects the target repo layout.

## Notes For Agents

- `SETUP_PROCESS.md` is the source of truth for creating a project.
- If the target repository is not empty, stop and ask for an empty one.
- Keep the flatpack repo focused on project assembly, not app implementation.
- Do not add target setup steps that depend on a local `.fssstack` directory. Target setup should read source kit files from doctrine with `dx read`.
- This root `AGENTS.md` is for agents working on the flatpack repo. It should not be copied to the root of the target repo.
- Target repo `AGENTS.md` is created during setup by following the doctrine standard named in `SETUP_PROCESS.md`.
