# Target Project Shape

This document gives the general shape of a generated fssstack target repo. Keep it high level so it stays useful as layers and tooling evolve.

Use this when changing `SETUP_PROCESS.md`, extension markdown, layer files, or setup scripts. The important question is always: what does the target repo look like at the moment this instruction runs?

## Mental Model

The flatpack Doctrine payload is not the application. It is the published source kit used to assemble an application somewhere else.

The target repo starts empty. Setup gradually unfolds it into a pnpm monorepo by reading files and scripts from doctrine with `dx read`, running a few external generators, rendering project-specific values, installing dependencies, and validating the generated result.

Do not write target setup instructions that assume:

- the flatpack repo exists inside the target repo
- a local `.fssstack` directory exists
- all apps and packages exist from the beginning
- dependencies are installed before the dependency stage
- template values have been rendered before the render stage

## Stages

### Empty Target

The agent starts in an empty target directory. At this point there should be no project files. If the target is not empty, setup should stop and ask for an empty repo.

### Foundation

The foundation stage creates the root workspace shape: package manager config, shared TypeScript/Vitest/ESLint config, Zap config, environment defaults, doctrine config, and other root files.

After this stage, the repo is recognizable as a workspace, but it is not a complete app. App directories, package directories, installed dependencies, and rendered project values may still be missing.

### Apps And Packages

The apps/packages stage adds shared packages and backend apps based on the service names chosen for the project. Each backend app includes a tRPC procedure integration-test helper at `src/test/createTestTrpc.ts` and a starter `*.procedure.test.ts` example so future backend behavior tests have an obvious home.

Instructions should not assume a single backend named `backend`. The examples may use that name, but real generated projects can have multiple backend services with user-chosen names.

### Frontends

Frontend apps are partly generated by external tooling and partly patched by fssstack layers.

For React/Vite and React/Next.js clients, the shadcn CLI owns the generic framework, React, Tailwind, and shadcn/ui scaffold. The fssstack overlay should stay narrow: workspace wiring, package scripts, shared config, deploy-safe hosting metadata such as Vite SPA rewrites, cleanup of scaffold noise, template-ready metadata, and a blank app surface.

Instructions should not bake product behavior, tRPC queries, or app-specific frontend tests into the generated scaffold. Use the frontend example markdown as guidance when creating the first real target app surface.

Instructions should not assume a single frontend named `frontend`. Real generated projects can have multiple clients with user-chosen names.

### Publishable Packages

Library packages live under `packages/<slug>` alongside the fixed internal packages. They are normal TypeScript workspace packages with package names rendered from the project package prefix and slug.

Library packages use the base tsconfig unless the target app needs a more specific runtime. A library that ships an executable just adds a package `bin` and builds an executable `dist/index.js` — it's still a library. If any publishable package exists, setup installs the shared Changesets direct-release workflow; a pushed changeset on `main` releases without a release PR.

### Template Rendering

Template rendering is the point where generic layer files become project-specific. This stage fills in project names, package names, descriptions, service lists, ports, app titles, favicon data, env values, and similar placeholders.

Before this stage, markdown and scripts should tolerate template tokens still being present. After this stage, validation should confirm generated values match the chosen project inputs.

### Dependencies

The dependency stage installs workspace dependencies, adds generated UI components where needed, normalizes package versions, and runs formatting or autofix commands used by setup.

Commands that require installed packages belong here or later. Do not place them in earlier stages unless they are run through `pnpm dlx`, `npx`, or another explicit one-off runner.

### Target Instructions

The target repo gets its own `AGENTS.md` during setup. That file is written for future work inside the generated application repo.

The authoring repo root `AGENTS.md` is only for agents editing this combined workspace. Do not copy its assumptions into the target repo.

### Validation

Validation happens from the target repo, after setup has produced the generated monorepo.

The goal is not to prove the flatpack repo is an app. The goal is to prove the generated target repo can lint, typecheck, test, build its apps/packages, start its services, and serve the expected health/frontend behavior.

## Implications For Changes

When changing setup markdown, be explicit about the stage where the instruction runs and the paths that exist at that stage.

When changing script behavior, edit `authoring-tools/scripts-src/`, run `pnpm -C authoring-tools build:docs`, and inspect the generated `.mjs` artifacts in the Doctrine payload folders. Target repos execute the built scripts through `dx read`.

When changing layer files, think about whether they are copied directly, applied as overlays, rendered later, or produced by an external generator. Keep fssstack overlays focused on project-specific wiring instead of taking ownership of generic scaffold code.

When changing extension docs, state whether the extension is applied during initial setup or after the base project validates. Use target repo paths and commands, not flatpack-local paths.
